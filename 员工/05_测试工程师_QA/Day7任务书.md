## 你要优先使用的 skills
- `pytest-patterns`
  用途：设计和补充 pytest 回归用例、边界测试、参数化验证。
- `github-actions`
  用途：如果你认为当前测试流程适合进一步标准化，可整理后续 CI 接入建议。

## 你的 Day 7 具体任务

### 任务 1：执行系统化回归测试
请先基于当前仓库执行并记录：
- `python -m pytest`
- `python -c "import app; print('app import ok')"`

然后围绕以下范围做回归验证：
- 趋势分析
- 对比分析（同比 / 环比）
- 留存分析
- 页面交互与状态切换

### 任务 2：验证前端交互闭环
请重点验证：
- 分析类型切换是否稳定
- SQL 方言切换是否稳定
- 动态表单是否能正确收集参数
- 示例加载是否正确回填参数
- 历史回看 / 清空历史是否正常
- 复制 / 下载是否具备基本可用性
- 参数摘要与结果展示是否一致

### 任务 3：验证边界与错误场景
请重点覆盖：
- 缺失必填字段
- 非法表名 / 字段名
- 不完整筛选条件
- 不合法日期范围
- 留存天数非法值
- 未开放分析类型（漏斗 / RFM）是否有明确提示

### 任务 4：补充测试代码
如发现当前自动化覆盖不足，请优先在 `tests/` 下补充：
- 页面辅助逻辑测试
- 参数校验测试
- 关键回归测试

请保持测试目的清晰，不要把实现逻辑复制到测试里。

### 任务 5：输出缺陷清单与回归建议
你需要给出明确结论：
- 哪些问题是阻塞演示的
- 哪些问题可以延期到后续阶段
- 哪些模块最值得优先修复

### 任务 6：提交完成总结
任务完成后，你必须在自己的文件夹下生成：
- [员工/05_测试工程师_QA/Day7完成总结.md](/E:/sql-generator/员工/05_测试工程师_QA/Day7完成总结.md)

完成总结至少要包含：
- 本次验证了哪些文件和能力
- 执行了哪些命令与结果
- 发现了哪些缺陷或确认了哪些通过项
- 当前仍未覆盖的风险点
- 推荐下一个对接人是谁以及原因

## 你交付时必须满足的验收标准
- 对当前三类已开放分析有明确测试结论
- 对页面核心交互有明确测试结论
- 对高风险边界场景有明确测试结论
- 若发现问题，输出可复现、可跟踪的缺陷清单
- 已在个人文件夹生成 `Day7完成总结.md`

## 你现在不要做的事
- 不要直接修改产品需求范围
- 不要把问题模糊地写成“可能有问题”
- 不要跳过高风险逻辑，尤其是留存分析和错误提示
- 不要替开发岗位直接实现一大批修复，而不先给出测试结论

现在项目已经有真实 SQL，也有可操作页面了。你这一轮最重要的是把“表面可用”变成“经过验证”。你的缺陷清单和回归建议，会直接决定后面技术负责人、SQL 工程师和前端工程师的修复优先级。
